What do we do that works? (5/x)

  • Keep everything close together. Eschew extra logins (to another project tracker, to CI) etc- reduces onboarding issues => more contributors.
  • A big community of technically capable people who can respond to urgent needs even if they don’t otherwise contribute often (and a way to reach them)
  • Encourage no-github-issue PRs for adding testing, upgrading deps, documentation, other dev chores (AND make chore issues so people know what needs doing)
  • Checklist for deploying + Checklist includes stakeholder communications
  • Weekly deploys, even if there is very little to deploy - keeping a cadence is key- so if you need to skip a week, it’s not a big deal

What do we do that works? (6/x)

Use available tooling!

  • Basics: PR templates, Issue templates (multiple & custom)
  • New issues auto-assign to the github project
  • Free CI however we can get it (farewell Travis, hello Github Actions)
  • Split CI into multiple jobs
  • Github bot to auto-tag with what's in the PR (ruby, js, db) for easier reviewing prioritzation
  • (controversial) Fast PR merge!!! Draws in contributors, keeps them engaged, and encourage follow-up PRs
  • To enable fast merge, qa -> staging -> prod 3 envs for safety (+ extra db/data migration testing env)
New issues auto-assign to the github project - trial and error, never found the docs on it, but url makes sense

A note on purpose

Be aware that your purpose as a project (may) be more than making a working tool for some users. Our purpose is also to help people learn how to contribute, and I consider it equally important. I could theoretically (for a while at least) build it all myself but that’s not what I want (time, flexibility, mental health) This may not be you! If you are big/popular project with more contributors than you want & a pack of paid maintainers, then this is not your problem (but turning the contributors that you have *into* the contributors that you *want* can work!) If you want more contributors than you have, you have to consciously and conspicuously invest in making sure that people know what to do. Office hours for people who want a human face, slack reminders for people who want to show up and figure it out, a good readme so people *can* figure it out, issues that explain what needs doing using url&screenshot&login rather than domain terms that new contributors won’t know
